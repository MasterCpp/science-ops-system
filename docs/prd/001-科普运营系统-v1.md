# PRD: 科普运营系统 V1

Status: `ready-for-agent`

## Problem Statement

科普活动运营需要同时管理活动发布、观众报名、现场签到、志愿者岗位和服务时长、来访报备、满意度问卷、活动照片归档以及多类数据导出。当前如果依赖人工表格、微信群、二维码表单和线下统计，数据容易分散，现场签到和志愿者服务时长难追溯，活动结束后整理报名表、签到表、志愿者表、问卷结果和总结数据会消耗大量人工时间。

项目还承担学习目标：以真实企业项目流程从 0 开始推进，形成可被多个 agent 对话复用的项目文档、PRD、架构决策、数据库设计、API 设计、测试和交付材料。

## Solution

建设一个第一版 Web 科普活动运营管理系统。后台管理端负责活动、报名、签到、志愿者、来访报备、问卷、照片、导出和日志管理；手机 H5 页面负责观众报名、观众签到、志愿者报名、志愿者签到/签退和问卷提交。

第一版采用“方案 B”的标准定制 Web 系统边界：后台管理端 + 移动 H5 + 后端服务 + MySQL 数据库 + 本地文件存储 + Docker/Nginx 部署。第一版不做微信小程序、公众号自动发布、短信验证码、复杂低代码表单设计器、固定 Logo/盖章导出模板或正式志愿者服务证明模板。

## User Stories

1. As a super admin, I want to create and manage back-office accounts, so that different staff can access the system with appropriate permissions.
2. As a super admin, I want passwords to be stored securely, so that account credentials are not saved in plain text.
3. As a super admin, I want to view operation logs, so that important administrative actions can be audited.
4. As a super admin, I want to delete activities when necessary, so that invalid or test activities can be removed by the highest-privilege role.
5. As an activity admin, I want to create an activity with title, cover, description, time, location, capacity, deadline, owner, phone, plan, process, and status, so that the activity has enough information for registration and management.
6. As an activity admin, I want to save an activity as draft, so that I can prepare information before publishing.
7. As an activity admin, I want to publish an activity for registration, so that audience users can register through a link or QR code.
8. As an activity admin, I want to move an activity through draft, registration open, in progress, ended, and archived states, so that activity operations follow a clear lifecycle.
9. As an activity admin, I want to edit all activity fields while the activity is draft, so that preparation remains flexible.
10. As an activity admin, I want changes during registration open to be logged when registration fields, capacity, or deadline change, so that sensitive changes are traceable.
11. As an activity admin, I want in-progress activities to lock registration fields and capacity, so that live activity data stays stable.
12. As an activity admin, I want ended activities to allow photos, attachments, and summary data to be added, so that post-event archive work can continue.
13. As a super admin, I want archived activities to be read-only unless unarchived by me, so that historical records are protected.
14. As an audience user, I want to open a mobile H5 registration page without creating an account, so that registration is quick.
15. As an audience user, I want to register with name, phone, attendee count, unit or school, age group, and remarks, so that the organizer has the needed basic information.
16. As an audience user, I want to submit custom fields configured for the activity, so that special activity-specific information can be collected.
17. As an audience user, I want registration to succeed immediately after submission, so that I do not need to wait for manual review.
18. As an audience user, I want to be blocked from registering after the registration deadline, so that the activity deadline is enforced.
19. As an audience user, I want to be blocked from registering when the activity capacity is full, so that capacity limits are respected.
20. As an audience user, I want the same phone number to register only once for the same activity, so that duplicate records are avoided.
21. As an activity admin, I want to backfill a registration after the deadline, so that offline or special-case attendees can still be recorded.
22. As an activity admin, I want to cancel a registration record, so that invalid registration data can be corrected.
23. As an activity admin, I want to view registration data, so that I can monitor attendance preparation.
24. As an activity admin, I want to export the registration table, so that I can share or archive attendee data.
25. As an audience user, I want to scan a fixed activity check-in QR code, so that I can complete check-in at the venue.
26. As an audience user, I want repeated scans to show that I have already checked in, so that duplicate check-in records are not created.
27. As an audience user, I want check-in to be allowed while the activity is in progress, so that on-site attendance can be captured.
28. As an audience user, I want mobile check-in to be blocked after the activity has ended, so that late invalid check-ins are prevented.
29. As an activity admin, I want to manually check in an attendee, so that exceptional on-site cases can be handled.
30. As an activity admin, I want to undo an attendee check-in, so that incorrect check-ins can be corrected.
31. As an activity admin, I want to export the check-in table, so that attendance records can be archived.
32. As a volunteer admin, I want to create volunteer positions for an activity, so that volunteer work can be organized by role.
33. As a volunteer admin, I want each position to have name, description, capacity, service start time, and service end time, so that volunteers understand the work and limits.
34. As a volunteer applicant, I want to apply for one position in an activity with name, phone, unit or school, age group, availability, experience, and remarks, so that I can volunteer for the activity.
35. As a volunteer applicant, I want to be blocked from applying to multiple positions in the same activity, so that assignment stays unambiguous.
36. As a volunteer applicant, I want to be blocked when a position is full, so that position capacity is enforced.
37. As a volunteer admin, I want to approve or reject volunteer applications, so that only confirmed volunteers can serve.
38. As a volunteer applicant, I want to check in only after approval, so that unapproved applicants cannot accumulate service time.
39. As a volunteer applicant, I want to check in and check out once for my approved position, so that service time can be calculated.
40. As a volunteer admin, I want to manually backfill, undo, or correct volunteer check-in and check-out, so that exceptional service records can be fixed.
41. As a volunteer admin, I want service hours to default to check-out time minus check-in time, so that service statistics are calculated consistently.
42. As a volunteer admin, I want to manually correct service hours, so that special cases can be reflected accurately.
43. As a volunteer admin, I want to export volunteer service data to Excel, so that volunteer work can be summarized.
44. As an activity admin, I want to create a visitor report with visitor unit, contact name, phone, visitor count, visit date, reason, related activity, and remarks, so that external visits can be tracked separately from audience registration.
45. As an activity admin, I want visitor reports to optionally relate to an activity, so that both activity-based and independent visits are supported.
46. As an activity admin, I want to export visitor reports, so that visitor data can be archived.
47. As an activity admin, I want to write activity plans in rich text and upload attachments, so that plans can include structured notes and supporting files.
48. As an activity admin, I want to manage activity process timeline entries, so that the event agenda is clear.
49. As an activity admin, I want attachments to support PDF, Word, Excel, PPT, and images up to 20MB, so that common planning materials can be stored.
50. As an activity admin, I want to upload activity photos in JPG, JPEG, PNG, or WEBP up to 10MB each, so that post-event photos can be archived.
51. As an activity admin, I want to preview photos in the back office, so that I can inspect uploaded activity images.
52. As an activity admin, I want to delete individual photos, so that incorrect photos can be removed.
53. As an activity admin, I want to batch download activity photos as a ZIP, so that archives can be handed off or stored offline.
54. As an activity admin, I want to create one satisfaction survey per activity, so that feedback is tied to a specific event.
55. As an activity admin, I want survey questions to support single choice, multiple choice, rating, and text, so that common satisfaction feedback can be collected.
56. As a checked-in audience user, I want to submit the satisfaction survey once, so that I can provide feedback after attending.
57. As an audience user, I want survey submission to be blocked if I have not checked in, so that only actual attendees submit feedback.
58. As an audience user, I want submitted surveys to be final, so that survey statistics remain stable.
59. As an activity admin, I want to view survey statistics and raw answers, so that I can understand feedback.
60. As an activity admin, I want to export survey results, so that feedback can be analyzed outside the system.
61. As an activity admin, I want an activity summary data page, so that I can quickly see registration count, check-in count, check-in rate, volunteer counts, service hours, survey count, average rating, and photo count.
62. As an activity admin, I want to export registration, check-in, volunteer, survey, and visitor data with confirmed columns, so that handoff data is consistent.
63. As a super admin, I want important operations to be logged, so that account, activity, export, check-in, volunteer review, and file deletion actions are traceable.
64. As a delivery owner, I want PRD, requirement clarification, ADRs, ER/database design, API docs, test cases, deployment docs, operation instructions, and acceptance checklist, so that the project can be developed and handed off like an enterprise system.

## Implementation Decisions

- Build V1 as a Web system with back-office management and mobile H5 pages.
- Use three back-office roles: super admin, activity admin, and volunteer admin.
- Mobile audience and volunteer users do not have accounts in V1; mobile flows identify users through activity links/QR codes and phone-based registration records.
- Use the activity lifecycle: draft, registration open, in progress, ended, archived.
- Activity creation uses the standard field set: title, cover, description, start time, end time, location, capacity, registration deadline, owner, contact phone, plan, process, and status.
- Registration uses fixed base fields plus limited activity-specific custom fields.
- Audience registration is immediately successful, unique per activity phone number, deadline-bound, and capacity-bound with no waitlist.
- Admins can backfill and cancel registration records according to the permission matrix.
- Each activity has a fixed check-in QR code.
- Audience check-in is one-time per registration, available while the activity is in progress, and can be manually backfilled or undone by permitted admins.
- Volunteers are managed per activity position, not through a full V1 volunteer profile library.
- Volunteer positions have capacity and service time windows; one volunteer can apply to only one position per activity; no volunteer waitlist in V1.
- Volunteer applications require review before check-in.
- Volunteer service hours default to check-out minus check-in and can be manually corrected by permitted admins.
- Visitor reports are independent records that may optionally link to an activity.
- Activity plans use rich text and attachments; activity process is represented as timeline entries.
- One satisfaction survey is supported per activity in V1.
- Survey answers are real-name, linked to registration phone/name, one submission per activity phone number, final after submission, and only available after check-in.
- File storage is local server storage in V1.
- Attachments support PDF, Word, Excel, PPT, and images up to 20MB.
- Photos support JPG, JPEG, PNG, and WEBP up to 10MB, with preview, single deletion, and activity ZIP download.
- Exports are Excel-based in V1; fixed Logo/header/seal templates are out of scope.
- Activity summary data page includes registration count, check-in count, check-in rate, volunteer application count, approved volunteer count, total volunteer service hours, survey submission count, average satisfaction rating, and photo count.
- Back-office authentication uses account and password; passwords must be stored encrypted/hashed.
- Operation logs cover login, activity create/update/delete, activity publish/end/archive, exports, registration backfill/cancel, check-in backfill/undo, volunteer review, and photo/attachment deletion.
- Phone numbers are fully visible to roles with permission in V1; no masking in V1.
- V1 includes database backup instructions but not automatic backup.
- Backend stack is Spring Boot, locked by ADR 001.

## Testing Decisions

- Test external behavior through feature-level flows rather than internal implementation details.
- Highest-value tests should cover complete business journeys: activity creation to publish, audience registration to check-in, volunteer application to service-hour calculation, survey submission after check-in, and export generation.
- Permission tests should verify that each role can only access the operations allowed by the confirmed permission matrix.
- State lifecycle tests should verify what can be edited in draft, registration open, in progress, ended, and archived states.
- Registration tests should cover deadline blocking, capacity blocking, duplicate phone blocking, admin backfill, and admin cancellation.
- Check-in tests should cover duplicate scan behavior, ended-activity blocking, manual backfill, and undo.
- Volunteer tests should cover position capacity, one-position-per-activity, review requirement, check-in/check-out, manual corrections, and export data.
- Survey tests should cover supported question types, checked-in-only submission, one submission per phone, no edit after submit, statistics, and export.
- File tests should cover allowed formats, size limits, photo preview, single deletion, and ZIP download.
- Export tests should verify required columns for registration, check-in, volunteer, survey, and visitor report exports.
- Operation log tests should verify that configured sensitive operations produce auditable log records.

## Out of Scope

- WeChat mini program.
- Official account automatic publishing.
- Forced binding to an outsourced private service.
- SMS verification.
- Complex low-code form designer.
- Volunteer waitlist.
- Audience waitlist.
- Full volunteer profile library.
- Audience check-out.
- Formal volunteer service certificate template.
- Fixed organization Logo, header, or seal position export templates.
- Phone number masking.
- Automatic database backup.
- Photo recycle bin or restore flow.
- Object storage integration.
